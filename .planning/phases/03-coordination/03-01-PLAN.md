  <name>Task 1: Intents schema + migration + declare/update/query RPC handlers</name>
  <files>src/db/schema.ts, src/daemon/rpc/handlers.ts, src/daemon/rpc/handlers.test.ts</files>
  <behavior>
    - intent.declare with {sessionId, description, files, symbols?} inserts intent row and returns {intentId}
    - intent.declare with symbols specified populates startByte/endByte by running extractSymbols on the named files (reuse existing parser pattern from lock.acquire)
    - intent.declare with no symbols sets startByte/endByte to null (file-level intent)
    - intent.declare stores files as comma-delimited text with leading/trailing commas for exact LIKE matching (e.g. ",src/auth.ts,src/utils.ts,") per research recommendation
    - intent.update transitions status: declared->active, declared->resolved, declared->abandoned, active->resolved, active->abandoned are valid
    - intent.update rejects invalid transitions (resolved->active, abandoned->active) with INVALID_TRANSITION error
    - intent.update sets updatedAt timestamp on each transition
    - intent.query with no filters returns all declared/active intents
    - intent.query with sessionId filter returns only that session's intents
    - intent.query with file filter matches intents touching that file (uses LIKE on comma-delimited files column)
    - intent.query with status filter returns only intents with that status
  </behavior>
  <action>
1. Add `intents` table to `src/db/schema.ts` following the schema from research:
   - `id` text primary key (crypto.randomUUID)
   - `sessionId` text not null
   - `description` text not null
   - `files` text not null (comma-delimited with leading/trailing commas: ",file1,file2,")
   - `symbols` text not null default "[]" (JSON array)
   - `startByte` int nullable, `endByte` int nullable
   - `status` text not null default "declared"
   - `declaredAt` int timestamp_ms not null, `updatedAt` int timestamp_ms not null
   - Indexes on sessionId, status, files

2. Run `bunx drizzle-kit generate` to create the migration SQL file.

3. Add three Zod schemas to handlers.ts:
   - `IntentDeclareParamsSchema`: sessionId (string.min(1)), description (string.min(1)), files (array of string.min(1), min length 1), symbols (optional array of string)
   - `IntentUpdateParamsSchema`: intentId (string.min(1)), sessionId (string.min(1)), status (enum: "active", "resolved", "abandoned")
   - `IntentQueryParamsSchema`: sessionId (optional string), file (optional string), status (optional string)

4. Add `intent.declare` case to handleRpc switch:
   - Parse byte ranges: if symbols are provided, for each file in files, detect language, read file content via Bun.file (resolve with WIT_REPO_ROOT/cwd like parseFileAndRefreshDeps), run extractSymbols, find matching symbol names, compute union range (min startByte, max endByte). If file doesn't exist or symbol not found, leave byte range null.
   - Insert intent row with crypto.randomUUID() id, comma-delimited files (with leading/trailing commas), JSON.stringify(symbols ?? []), byte range, "declared" status, Date.now() timestamps.
   - Return `{intentId: id}`. Do NOT include conflict detection in this plan -- Plan 02 adds that.

5. Add `intent.update` case:
   - Fetch intent by id. If not found, return INTENT_NOT_FOUND error.
   - Verify sessionId matches intent's sessionId. If not, return INTENT_NOT_OWNED error.
   - Validate transition: define valid transitions map: {declared: ["active","resolved","abandoned"], active: ["resolved","abandoned"]}. If current status not in map or target status not in allowed list, return INVALID_TRANSITION error with {current, requested}.
   - Update status and updatedAt. Return {intentId, status, updatedAt}.

6. Add `intent.query` case:
   - Build Drizzle where conditions dynamically: always filter status in ["declared","active"] unless status param provided.
   - If sessionId provided, add eq filter.
   - If file provided, add `sql\`${intents.files} LIKE ${'%,' + file + ',%'}\`` for exact segment matching in comma-delimited column.
   - If status provided, filter by that exact status (allows querying resolved/abandoned intents too).
   - Return array of intent objects with all fields.

7. Write tests in handlers.test.ts extending the existing describe blocks:
   - Test intent.declare inserts and returns intentId
   - Test intent.declare with symbols populates byte range (requires real ParserService fixture, same setup as lock.acquire caller warning tests)
   - Test intent.update valid transitions
   - Test intent.update invalid transition returns error
   - Test intent.update wrong session returns INTENT_NOT_OWNED
   - Test intent.query returns active intents
   - Test intent.query filters by sessionId
   - Test intent.query filters by file
