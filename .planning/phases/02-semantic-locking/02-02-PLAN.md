<task type="auto" tdd="true">
  <name>Task 1: Lock acquire/release/query RPC handlers with tests</name>
  <files>src/daemon/rpc/handlers.ts, src/daemon/rpc/handlers.test.ts</files>
  <behavior>
    - lock.acquire with valid symbolPath + sessionId + ttlMs inserts lock row and returns { symbolPath, sessionId, acquiredAt, expiresAt }
    - lock.acquire with default TTL (no ttlMs param) uses 1_800_000ms (30 min)
    - lock.acquire on symbol already locked by SAME session upserts (refreshes expiresAt) and returns success
    - lock.acquire on symbol locked by DIFFERENT session returns RpcError code -32000 message "LOCK_CONFLICT" with data { heldBy, expiresAt }
    - lock.acquire with missing/invalid params returns INVALID_REQUEST
    - lock.release with valid symbolPath + sessionId deletes lock row and returns { released: true }
    - lock.release on symbol not locked returns RpcError code -32000 message "LOCK_NOT_FOUND"
    - lock.release on symbol locked by DIFFERENT session returns RpcError code -32000 message "LOCK_NOT_HELD" with data { heldBy }
    - lock.query with no filters returns all active (non-expired) locks with ttlRemainingMs computed
    - lock.query with sessionId filter returns only that session's locks
    - lock.query does not return expired locks
  </behavior>
  <action>
    1. Add zod schemas for lock RPC params in handlers.ts:
       - `LockAcquireParamsSchema`: { symbolPath: string (min 1, must contain ":"), sessionId: string (min 1), ttlMs?: number (optional, default 1_800_000) }
       - `LockReleaseParamsSchema`: { symbolPath: string (min 1), sessionId: string (min 1) }
       - `LockQueryParamsSchema`: { sessionId?: string (optional filter) }

    2. Add three new cases to the switch in handleRpc:

       **"lock.acquire":**
       - Parse params with LockAcquireParamsSchema
       - Compute expiresAt = new Date(Date.now() + (ttlMs ?? 1_800_000))
       - Query existing lock: `db.select().from(locks).where(eq(locks.symbolPath, symbolPath)).limit(1)`
       - If existing lock held by different sessionId AND not expired: return LOCK_CONFLICT error with { heldBy: existing.sessionId, expiresAt: existing.expiresAt }
       - Otherwise upsert: `db.insert(locks).values({...}).onConflictDoUpdate({ target: locks.symbolPath, set: { expiresAt, sessionId, acquiredAt: new Date() } })`
       - Return success with lock details

       **"lock.release":**
       - Parse params with LockReleaseParamsSchema
       - Query existing lock on symbolPath
       - If no lock exists: return LOCK_NOT_FOUND error
       - If lock held by different sessionId: return LOCK_NOT_HELD error with { heldBy }
       - Delete: `db.delete(locks).where(eq(locks.symbolPath, symbolPath)).run()`
       - Return { released: true }

       **"lock.query":**
       - Parse params with LockQueryParamsSchema
       - Build query: select all from locks where expiresAt > now
       - If sessionId filter provided, add where clause
       - Map results to include ttlRemainingMs = expiresAt.getTime() - Date.now()
       - Return array of lock objects

    3. Import `eq`, `gt`, `and` from "drizzle-orm" and `locks` from schema.

    4. Tests: Use the same test pattern as existing handlers.test.ts -- create in-memory DB, run migrations, call handleRpc directly.
       - For each test, create a fresh DB and construct deps, then call handleRpc with appropriate RpcRequest objects
       - Test conflict scenario: acquire lock as session-A, then attempt acquire as session-B on same symbol
       - Test idempotent re-lock: acquire as session-A, acquire again as session-A with new TTL
       - Test expired lock not blocking: insert lock with expiresAt in the past, verify new acquire succeeds
  </action>

  </verify>
  <done>
    - lock.acquire creates locks, handles conflicts, supports idempotent re-lock
    - lock.release deletes locks with ownership check
    - lock.query returns active locks with computed TTL remaining
    - All handler tests pass including new lock method tests
  </done>
